classical machine learning, and transformer models into a multi-tier ensemble
system for content moderation.

2.3 REQUEST FLOW
----------------

1. Client sends POST /score with {"text": "content"}
2. FastAPI receives request → Rate limiting check
3. Request logged with unique trace_id
4. Ensemble guard selects detection method:
   a. Quick texts (<50 chars) → Rules only
   b. Medium texts (50-300 chars) → Rules + ML
   c. Complex texts (>300 chars or obfuscated) → Full ensemble
5. Detection performed (parallel execution when possible)
6. Result aggregated with confidence scores
7. Policy headers added (X-Policy-Version, X-Policy-Checksum)
8. Response returned with prediction + metadata
9. Metrics recorded (latency, guard method, category)

3.4 ENSEMBLE GUARD
------------------

File: src/guards/ensemble_guard.py
Strategy: Intelligent routing with weighted voting
Latency: 2.5ms average

Routing Logic:
  1. All texts → Rules engine (fast baseline)
  2. If rules flag → Return immediately
  3. If rules pass:
     a. Text length < 50 chars → Skip ML
     b. Text length 50-300 → Run ML
     c. Text length > 300 or obfuscated → Run all

Voting Strategy:
  • Rules + ML agree (flag) → flag (confidence: 0.85)
  • Rules + Transformer agree (flag) → flag (confidence: 0.90)
  • All three agree (flag) → flag (confidence: 0.95)
  • All three agree (safe) → pass (confidence: 0.95)
  • Split decision → Weighted average of confidences

Decision Thresholds:
  • Flag threshold: 0.5 (configurable)
  • High confidence: > 0.8
  • Uncertainty range: 0.4-0.6

Sexual:

  • Sexual violence
  • Examples: "send nudes", "rape", "child exploitation"

Harassment:
  • Identity-based attacks
  • Dehumanization
  • Doxxing threats
  • Slurs and hate speech
  • Examples: racial slurs, "kill all X", doxxing threats

Illegal:
  • Drug trafficking
  • Weapons sales
  • Fraud/scams
  • Examples: "buy cocaine", "fake ID", "credit card fraud"

4.2 OBFUSCATION DETECTION
--------------------------

Techniques Detected:
  • Leetspeak: k!11 → kill, d13 → die
  • Character substitution: @ → a, 3 → e, 1 → i, 0 → o
  • Zero-width characters: word\u200bspacing
  • Homoglyph attacks: а (Cyrillic) vs a (Latin)
  • Excessive punctuation: k.i.l.l
  • Emoji encoding: 🔫💀☠️
  • Unicode abuse: ｋｉｌｌ (fullwidth)

Implementation:
  • Shannon entropy calculation (detect random patterns)
  • Unicode normalization (NFKC)
  • Character frequency analysis
  • Pattern matching with tolerance

4.3 CONTEXT AWARENESS
---------------------

Safe Contexts (Not Flagged):
  • Medical: "cancer treatment", "pain management"
  • Media: "movie violence", "game shooting"
  • Education: "history of war", "anatomy class"
  • Idioms: "killing time", "dead serious"

Context Signals:
  • Presence of safe keywords
  • Sentence structure analysis
  • Surrounding context (if available)
  • Domain-specific terminology

10. FEDERATION NETWORK (AEGIS)
================================================================================

10.1 ARCHITECTURE
-----------------

Purpose: Privacy-preserving threat intelligence sharing

Components:
  • Antibody Storage: data/federation.yaml
  • Reputation System: Weighted voting
  • Gossip Protocol: REST API-based
  • Privacy: Only patterns shared, never raw text

10.2 ANTIBODY LIFECYCLE
-----------------------

1. Detection: Novel attack pattern detected
2. Generation: Regex pattern created (scripts/aegis_pattern_generator.py)
3. Quarantine: Pattern stored in quarantine list
4. Submission: POST /federation/intel-exchange
5. Validation: Reputation score calculated
6. Distribution: GET /federation/export by other nodes
7. Voting: Upvote/downvote by community
8. Pruning: Expired (>30 days) or low-rep (<0.3) removed

10.3 REPUTATION SCORING
-----------------------

Formula:
  reputation = (accuracy * 0.7) + (vote_score * 0.3)

Where:
  accuracy = 1 - (false_positives / total_seen)
  vote_score = upvotes / (upvotes + downvotes)

Thresholds:
  • Minimum for distribution: 0.6
  • Auto-prune: <0.3 with 10+ seen
  • High confidence: >0.8

10.4 PRIVACY GUARANTEES
-----------------------

Shared:
  • Regex patterns
  • Confidence scores
  • Category labels
  • Accuracy metrics
  • Anonymized node IDs

NOT Shared:
  • Raw user text
  • User IDs
  • IP addresses
  • Business logic
  • Custom rules

Compliance: GDPR/CCPA compliant by design

13.1 HORIZONTAL SCALING
-----------------------

Strategy: Stateless application design

Load Balancing:
  • Algorithm: Round-robin or least connections
  • Health Checks: GET /healthz every 30s
  • Unhealthy Threshold: 3 consecutive failures

Session Affinity: Not required (stateless)

Auto-Scaling Triggers:
  • CPU > 70%
  • Memory > 80%
  • Request queue > 100

Scale-Up: Add instances (1-2 minutes)
Scale-Down: Remove instances (5 minutes cooldown)

13.2 CACHING STRATEGY
---------------------

Content Caching:
  • Key: SHA256(text + policy_version)
  • TTL: 1 hour
  • Hit Rate: 90%+ for repeated content
  • Backend: Redis (optional, in-memory default)

Model Caching:
  • ML model: Loaded once at startup
  • Transformer: Loaded on first use
  • Policy: Cached with mtime check

Response Caching:
  • Enabled for identical requests
  • Invalidated on policy update

13.3 FAULT TOLERANCE
--------------------

Retry Logic:
  • Transient errors: 3 retries with exponential backoff
  • Timeout errors: 1 retry
  • Authentication errors: No retry

Circuit Breaker (Planned):
  • Threshold: 50% error rate in 10 seconds
  • Open duration: 30 seconds
  • Half-open: Allow 1 test request

Graceful Degradation:
  • Transformer unavailable → Use ML + Rules
  • ML unavailable → Use Rules only
  • Database unavailable → In-memory only

13.4 BACKUP & RECOVERY
----------------------

Database Backups:
  • Frequency: Daily (automated)
  • Retention: 30 days
  • Storage: Cloud object storage (S3/GCS/Blob)

Disaster Recovery:
  • RTO (Recovery Time Objective): 1 hour
  • RPO (Recovery Point Objective): 24 hours
  • Procedure: Documented in docs/RUNBOOKS.md

Model Versioning:
  • Models stored with version tags
  • Rollback: Switch to previous version

14.2 BATCH PROCESSING
---------------------

Recommended for: >100 texts

Python:
  texts = ["text1", "text2", "text3"]
  results = client.batch_score(texts)
  for result in results:
      print(f"{result.id}: {result.prediction}")

Best Practices:
  • Batch size: 50-100 items
  • Parallel batches: 5-10 concurrent requests
  • Timeout: 60 seconds

14.3 ERROR HANDLING
-------------------

HTTP Status Codes:
  • 200 OK: Success
  • 400 Bad Request: Invalid input
  • 401 Unauthorized: Missing/invalid API key
  • 429 Too Many Requests: Rate limit exceeded
  • 500 Internal Server Error: Server error
  • 503 Service Unavailable: Temporary unavailability

Python Example:
  try:
      result = client.score(text)
  except AuthenticationError:
      print("Invalid API key")
  except RateLimitError as e:
      print(f"Rate limited. Retry after {e.retry_after}s")
  except SeaReiError as e:
      print(f"Error: {e}")
